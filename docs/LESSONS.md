## 🛠 해결된 문제 (Solved Issues)

### 1. 객관식 보기의 무한 리셔플(Re-shuffle) 현상
- **증상**: 구구단 풀이 중 다른 이벤트(예: 마우스 호버, 미세한 상태 변화)로 인해 화면이 재렌더링될 때마다, 4지선다 보기 배열의 순서가 계속 무작위로 바뀌며 사용자가 선택하기 매우 힘들어지는 레이아웃 불안정 현상 발생.
- **원인**: JSX 렌더링 함수 영역 안에서 `generateMultipleChoices(a, b)`를 직접 리턴하게 코딩했기 때문. 리액트 컴포넌트는 상태 변화 시 렌더링 본문을 위에서 아래로 다시 훑으며 계산하므로, 내부의 셔플 함수가 무한 호출됨.
- **해결책**:
  ```javascript
  // 1. 보기를 담을 별도 상태 정의
  const [currentChoices, setCurrentChoices] = useState([]);

  // 2. 문제 인덱스가 바뀔 때만(또는 문제가 로드될 때만) 1회성 계산 후 캐싱
  useEffect(() => {
    if (questions.length > 0 && questions[currentIndex]) {
      const { a, b } = questions[currentIndex];
      setCurrentChoices(generateMultipleChoices(a, b));
    }
  }, [questions, currentIndex]);
  ```
- **배운 점**: UI에 바인딩할 데이터 중 **"비결정론적 속성(무작위 셔플 등)"**을 지닌 함수 연산 결과는 절대 렌더링 파이프라인 내부에서 직접 계산하면 안 됨. 반드시 상태(`state`)에 담아 단방향 1회 업데이트 후 고정 캐싱해서 렌더링해야 함.

### 2. 가상 키패드 상태 지연 입력 및 중복 클릭 이슈
- **증상**: 주관식 정답 입력 시 "확인" 혹은 "제출" 버튼을 연속으로 클릭하면, 정답 판정 연산이 중복 실행되어 점수가 한 번에 여러 개 올라가거나 문제 인덱스가 강제로 2칸씩 건너뛰어지는 버그 발생.
- **원인**: 비동기 트랜지션 타이머(`setTimeout` 850ms)로 다음 문제 전이를 대기하는 동안, 사용자가 계속 확인 버튼이나 가상 키 버튼을 연쇄 클릭할 수 있도록 입력 락(Lock) 장치가 없었기 때문.
- **해결책**:
  - `isCorrecting`이라는 락킹 상태 변수를 정의.
  - 정/오답 판정 함수 `submitAnswer`가 호출되는 순간 `setIsCorrecting(true)`로 락을 걸고, 가상 키패드 버튼과 확인 버튼의 `disabled={isCorrecting}`을 활성화하여 클릭 이벤트를 원천 차단.
  - 다음 문제 전이 `useEffect` 단에서 락을 해제하도록 설계.
- **배운 점**: 비동기 처리가 수반되는 사용자 인터랙션(트랜지션, 페이지 이동 등)은 반드시 프론트엔드 레벨에서 API 요청과 마찬가지로 **"멱등성 및 락킹(Locking) 장치"**를 두어 다중 클릭을 적극 예방해야 함.
